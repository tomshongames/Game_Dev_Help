Treat this as an automatic refactor task, not a teaching task.

I will provide a single Godot/GDScript script or pasted code block.

Process:
1. Inspect the full provided code before making changes.
2. Identify internal relationships between classes, functions, signals, constants, nodes, and references.
3. Determine what can be safely standardized or renamed.
4. Refactor only after understanding the script as a whole.

Naming rules:
- Use snake_case for variables, functions, parameters, signals, file names, and folder names.
- Use PascalCase for class names, scene names, and node names.
- Use UPPER_CASE for constants.

- Rename for clarity only when the current name is clearly too vague, misleading, a typo, or overly minimal.
- Prefer minimal necessary changes over aggressive refactoring.
- Preserve logic, behavior, intended output, structure, ordering, formatting style, indentation, and line endings unless a change is required for correctness or consistency.
- Do not restructure the code unnecessarily.
- Do not change execution order, node paths, scene structure, or project-facing behavior unless required to keep the code working.

- Preserve annotations, decorators, comments, and region markers unless an update is required because of a rename.
- Keep comments intact unless they become inaccurate after a rename.
- Do not remove unused code unless it is clearly broken or invalid.
- Fix only clear syntax errors, clearly broken references, or safe naming-related issues visible in the provided code.
- Respect the Godot version and syntax already used unless the code is clearly broken.
- Do not modernize syntax or APIs unless necessary for correctness.
- Do not invent missing nodes, scenes, resources, dependencies, or project structure.

Godot safety rules:
- If a rename could break behavior, editor links, external references, dynamic calls, project settings, autoloads, resources, scene integration, node paths, serialized data, or project structure, keep the original name unchanged.

- When unsure whether a rename is safe, keep the original name unchanged.
- When rules conflict, prefer preserving behavior and project integration over renaming for style.
