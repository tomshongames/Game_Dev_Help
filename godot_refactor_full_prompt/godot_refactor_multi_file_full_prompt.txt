Treat this as an automatic refactor and optimization task, not a teaching task.

I may provide multiple Godot/GDScript scripts, connected files, a folder structure with file contents, or part or all of a Godot project.

Your job is to inspect, refactor, and optimize the provided material so it follows strict Godot naming conventions and style rules, while preserving behavior, intended output, safe project integration, and visible project structure unless a clearly safe rename, cleanup, or optimization is possible.

CORE PRIORITY
- Behavior, project integration, compatibility, and visible project structure are more important than style or optimization.
- When safety is uncertain, keep the original unchanged.
- Never claim to have updated references that were not visible.

WORKFLOW
1. Inspect all provided code, scripts, scenes, files, folders, and paths before making changes.
2. Determine which scripts, scenes, classes, signals, constants, node references, autoloads, resources, strings, files, and folders are related.
3. Build an internal dependency map across the visible material only.
4. Detect cross-file dependencies, dynamic references, scene/script/resource links, editor-facing bindings, and project structure connections before applying changes.
5. Identify what can be safely renamed, standardized, cleaned up, de-duplicated, fixed, or optimized from the visible material only.
6. Apply a safe refactor pass first for naming, clarity, consistency, and visible correctness.
7. Only after the safe refactor pass is complete, apply safe optimizations and safe redundancy reduction that do not change behavior, timing, integration, visible structure, editor links, serialized data expectations, or runtime meaning.
8. After the applied changes, provide a separate opinion section with possible further improvements, remaining redundancies, and higher-risk optimizations that were not applied because they may alter behavior, structure, timing, balance, compatibility, or rely on incomplete visibility.

PRIMARY GOAL
- Refactor the provided material to match the style guide where clearly safe.
- Rename variables, functions, parameters, signals, classes, constants, file names, folder names, and internal identifiers whenever it is clearly safe.
- Apply safe optimizations that improve clarity, maintainability, consistency, redundancy reduction, or runtime efficiency without changing behavior.
- Keep all related visible references consistent after every safe rename or safe optimization.
- If a rename, fix, cleanup, deduplication, optimization, or structural improvement may break behavior, editor links, external references, dynamic calls, project settings, autoloads, resources, scene integration, node paths, serialized data, cross-file links, timing, gameplay balance, or project structure, do not apply it. Mention it later only as a suggestion if supported by the visible material.

NAMING RULES
- Use snake_case for variables, functions, parameters, signals, file names, and folder names.
- Use PascalCase for class names, scene names, and node names.
- Use UPPER_CASE for constants.

REFACTOR RULES
- Treat all provided files as potentially connected unless clearly independent.
- Maintain cross-file naming consistency across all visible material.
- Rename booleans to natural names such as is_dead, can_move, has_target, has_key, is_active, or can_attack when appropriate.
- Rename for clarity only when the current name is clearly too vague, misleading, typo-ridden, overly minimal, or inconsistent.
- Do not rename for subjective preference alone.
- Prefer minimal necessary changes over aggressive refactoring when safety is uncertain.
- Preserve original logic, behavior, execution flow, intended output, compatibility, and project-facing meaning.
- Do not restructure code, split files, merge files, move files, or reorganize folders unless explicitly requested or clearly required for correctness and proven safe from the visible material.
- Keep comments, annotations, decorators, disabled code, region markers, and intentional formatting unless a safe rename or safe correction requires an update.
- Do not remove unused code unless it is clearly broken, invalid, duplicated in a clearly safe way, or safely removable from the visible material.
- Fix only clear syntax errors, clearly broken visible references, naming-related issues, or clearly safe cleanup/performance issues that can be safely corrected from the visible material.
- Respect the existing Godot version and syntax style unless the visible code is clearly broken.

OPTIMIZATION RULES
- Optimize only when the optimization is clearly safe from the visible material.
- Prioritize, in order:
  1. clarity
  2. maintainability
  3. consistency
  4. redundancy reduction
  5. safe runtime efficiency
- Safe optimization examples include:
  - removing clearly redundant temporary values when behavior remains identical
  - reducing repeated lookups when the result is stable in the visible scope
  - simplifying duplicated visible logic when the outcome remains identical
  - caching repeated node or property access when clearly safe
  - tightening loops or conditions when behavior remains identical
  - removing obvious no-op code
  - fixing clearly inefficient visible patterns when the intended result is unchanged
  - improving local naming clarity that reduces maintenance risk
- Do not apply optimizations that may change:
  - timing
  - update order
  - physics behavior
  - signal timing
  - scene loading behavior
  - lazy vs eager loading semantics
  - memory lifecycle
  - serialization expectations
  - editor behavior
  - tool scripts
  - networking behavior
  - animation flow
  - state flow
  - randomness
  - input behavior
  - game balancing
- If an optimization seems beneficial but is not fully safe, do not apply it. Mention it later as a suggestion instead.

GODOT-SPECIFIC SAFETY RULES
- Never rename Input action strings or InputMap-related strings.
- Do not rename dictionary keys, JSON keys, config keys, save-data keys, animation names, state names, group names, bus names, shader parameter names, plugin/tool identifiers, or other string-based references unless all related visible references can also be safely updated.
- Do not rename nodes if they are referenced through $, %, get_node(), get_node_or_null(), NodePath, find_child(), find_node(), owner lookups, exported NodePaths, serialized scene data, or external resources unless the corresponding scene-tree names are also visible and can be safely updated everywhere relevant.
- Preserve @tool behavior and avoid renaming inspector-facing exported names when that could break scene files, editor bindings, tooling, or serialized data.
- Check dynamic references before renaming or optimizing anything, including call(), callv(), Callable(), has_method(), emit_signal(), emit(), RPC names, method-name strings, signal-name strings, node-name strings, animation-name strings, and state-name strings.
- Keep signal declarations, signal emissions, connections, and connected methods consistent after any safe rename.
- If unsure whether a rename, cleanup, fix, or optimization is safe, do not apply it.

PROJECT-AWARENESS RULES
- Use all provided file paths and folder paths to preserve project context.
- If multiple files are provided, check whether they reference each other before changing anything.
- If a file, folder, class, or scene name is safely renamed, update all clearly related visible references.
- If the material is incomplete, refactor and optimize only what is visible.
- Never assume unseen .tscn, .tres, .res, .godot, addon, plugin, autoload, or project settings files were updated unless they were explicitly provided.
- Never invent missing files or pretend unseen references were updated.
- Preserve cross-file compatibility over style or optimization whenever there is uncertainty.

FOLDER STRUCTURE STANDARD (SAFE-APPLY)

Target preferred structure:

scripts/
  *.gd

scenes/
  *.tscn

art/
  textures, sprites, images

audio/
  music, sfx

- Scripts should be placed inside "scripts/" when clearly safe.
- Scenes should be placed inside "scenes/" when clearly safe.
- Visual assets should be placed inside "art/" when clearly safe.
- Audio assets should be placed inside "audio/" when clearly safe.

- This rule is secondary to all safety rules:
  - Do NOT move files if any references (NodePaths, preload/load paths, resources, scenes, autoloads, or dynamic strings) are not fully visible.
  - Do NOT move files if it may break scene links, resource links, or project structure.
  - Do NOT assume unseen files were updated.

- Only apply file moves when ALL of the following are true:
  1. The file is clearly misplaced
  2. All visible references to that file can be safely updated
  3. No dynamic or string-based references are detected
  4. The move does not affect runtime behavior, loading, or editor bindings

- If safe file relocation is applied:
  - Update all visible paths accordingly
  - Preserve relative structure inside each folder
  - Do not rename files unless also clearly safe

- If structure is inconsistent but cannot be safely changed:
  - Keep original structure unchanged
  - Report it in "Opinion / Possible Further Improvements"

- Do NOT reorganize beyond this structure

- Prefer sub-organization inside folders when clearly safe:

  Examples:
  art/player/
  art/enemy/
  art/ui/
  audio/music/
  audio/sfx/

- Group assets by purpose or entity when:
  - All references are visible
  - Moving them will not break paths

- Do NOT enforce subfolder creation if:
  - Structure is already in use
  - References are not fully visible
  - It may break resource paths or scene links

- Naming inside folders:
  - Prefer descriptive names like:
    player_sprite.png, enemy_sprite.png, ui_button.png
  - Do NOT rename assets if they are referenced and not fully visible

FILE AND FOLDER RENAME POLICY
- File renames are higher risk than symbol renames.
- Folder renames are allowed when clearly safe.
- Only rename file names or folder names when all visible references to them are also visible and can be safely updated.
- If file or folder rename safety is uncertain, keep the original file or folder name unchanged.
- Do not move, split, merge, or reorganize files or folders unless explicitly requested or clearly required for correctness and proven safe from the visible material.

VERSION AND COMPATIBILITY RULES
- Respect the Godot version and syntax already used unless clearly broken.
- Do not modernize syntax or APIs unless required for correctness.
- Do not convert Godot 3 style to Godot 4 style, or vice versa, unless necessary and clearly supported by the visible material.
- Preserve typed vs untyped style unless a small safe correction is needed for consistency or correctness.
